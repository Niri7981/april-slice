# 2026-05-29 Development Log

Today I mainly read and organized the Initial Hand files.

## Main files

### `initialHand.ts`

This file defines the `InitialHand` structure and uses Zod to validate data returned by the interpreter.

The important point is that output from the LLM or fallback cannot be trusted directly. It must pass through `initialHandSchema.parse(output)` before it becomes a trusted `InitialHand`.

### `astrologyInitialHand.ts`

At first, I looked at this direction: generating an Initial Hand from astrology or birth-date information.

That idea was later split more clearly. Local code should not hard-code the final tags directly. Astrology information should enter an interpretation flow, and the result should be structured Initial Hand output.

### `initialHandPrompt.ts`

This file turns a `chart seed` into a prompt.

It does not generate the Initial Hand directly. Instead, it organizes the astrology data, game tone, output rules, and forbidden patterns into a prompt for the interpreter.

### `initialHandResolver.ts`

This file connects the process:

- Call the prompt builder.
- Call `interpret({ chart, prompt })`.
- Validate the returned result.

## Initial Hand data flow

```text
AstrologyChartSeed
        ↓
buildInitialHandPrompt({ chart })
        ↓
prompt string
        ↓
interpret({ chart, prompt })
        ↓
output / InitialHand candidate
        ↓
initialHandSchema.parse(output)
        ↓
trusted InitialHand
```

## Today's understanding

Initial Hand should not be hard-coded personality labels written directly into the code.

The better direction is:

```text
astrology data
  -> LLM / interpreter reading
  -> structured Initial Hand tags
  -> Zod validation
  -> run state and prompt
```

These tags should only be used as opening bias, tone, misreading tendency, and relationship interpretation bias. They should not directly decide behavior, numbers, or plot outcomes.
