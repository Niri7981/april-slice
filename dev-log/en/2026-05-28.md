# 2026-05-28 Development Log

Today I mainly started reading from `WorldPrototype`.

## WorldPrototype

The first important part is:

```ts
export function WorldPrototype() {
  const { state, actions } = useWorldRuntime();
}
```

`useWorldRuntime` is the hook for the world runtime. It returns two things: `state` and `actions`.

`state` is then split and passed to different parts of the UI.

For the diary, `diaryParagraphs` checks whether there is real diary content. If there is, it displays the real diary. If not, it falls back to the fallback text.

`traceSummary` joins the echo records together.

Then comes the `return` section. `Application` is the Pixi canvas environment. `WorldStage` is the component that actually draws and runs the world.

`WorldPrototype` is basically saying:

> I give you a canvas. You render the world inside it.

The ideal responsibility split looks like this:

```text
WorldPrototype
├── useWorldRuntime()       // gets world state and actions
├── Application             // Pixi canvas container
│   └── WorldStage          // world stage
├── NoteEchoDialog          // note UI
└── DiaryView               // diary UI
```

My current understanding is:

```text
useWorldRuntime = world brain
WorldPrototype = wiring board
WorldStage = world view
NoteEchoDialog = note interaction layer
DiaryView = diary interaction layer
```

The core data flow in `WorldPrototype`:

```text
useWorldRuntime
      ↓
state + actions
      ↓
WorldPrototype
      ↓
passes state to WorldStage / Dialog / Diary
passes actions to WorldStage / Dialog / Diary
      ↓
user interacts with UI
      ↓
actions are called
      ↓
runtime updates state
      ↓
React re-renders
```

## WorldStage

The next file I read was `WorldStage`.

Its responsibility: it is the “world stage” that actually goes inside the Pixi canvas.

It combines visual layers such as the map, note paper, NPCs, Agent, and Player, and it starts the world loop through `useWorldLoop(props)`.

```text
WorldStage
├── useWorldLoop(props)     // world loop, character positions, collision, time progression
└── pixiContainer           // Pixi stage container
    ├── WorldMapSprite      // map
    ├── NotePaperSprite     // note paper
    ├── NpcLayer            // NPC layer
    ├── AgentSprite         // Agent character
    └── PlayerSprite        // Player character
```

```ts
extend({ Container, Graphics, Text });
```

This registers Pixi components. In short, it enables the React-style Pixi syntax for `Container / Graphics / Text`.

`WorldStageProps` currently looks like this:

```ts
type WorldStageProps = {
  day: number;
  paused: boolean;
  noteAvailable: boolean;
  agentState: AgentSignalState;
  echoEffect: EchoBehaviorEffect | null;
  onNotePicked: () => void;
  onDayComplete: () => void;
  onWorldContextChanged: (scene: WorldNodeId, timeOfDay: WorldTimeOfDay) => void;
};
```

The core call is:

```ts
const { refs, snapshot } = useWorldLoop(props);
```

`WorldStage` does not implement movement logic, collision logic, or time progression itself. All of that is delegated to `useWorldLoop(props)`.

It returns two things:

- `refs`: references to Pixi objects, used to directly manipulate objects on the canvas.
- `snapshot`: the world snapshot for the current frame or current state.

The render order also matters. It is roughly:

```text
bottom layer: map
middle layer: note paper
middle layer: NPCs
upper layer: Agent
top layer: Player
```

The layer relationship can be understood like this:

```text
WorldPrototype
owns the React page layer:
- creates the Pixi Application
- manages overlays
- gets state/actions from useWorldRuntime
- passes props to WorldStage

        ↓

WorldStage
owns the Pixi stage layer:
- registers Pixi elements
- calls useWorldLoop
- renders map, note paper, NPCs, Agent, and Player

        ↓

useWorldLoop
owns the world loop:
- player movement
- agent movement
- time progression
- note pickup detection
- day completion detection
- scene context changes
```

## useWorldLoop

Then I read the `useWorldLoop` hook.

File responsibility: it is the core game-loop hook for this small world.

When `day` changes, it resets the whole world:

```ts
useEffect(() => {
  // reset world
}, [day]);
```

For example, when day one ends and day two begins, it needs to:

- return the player to the initial position
- return the agent to the initial route
- reset the camera
- reset world time to morning
- allow the day-end event to fire again
- clear the school pause state
- clear context tracking

`useTick` is the real game loop.

This is where context changes are reported:

```ts
if (lastContextKey.current !== worldContext.key) {
  lastContextKey.current = worldContext.key;
  onWorldContextChanged(worldContext.scene, worldContext.timeOfDay);
}
```

Context is reported to the runtime only when it changes.

The main flow of this file is: every frame, `useTick` runs:

1. If `paused`, stop immediately.
2. Calculate `dt`.
3. Advance the player and camera based on keyboard input.
4. Advance world time.
5. Generate world context from player position and time.
6. If context changed, notify the runtime.
7. Advance the agent based on time, agent state, and echo effect.
8. Check whether the player pressed `E` to pick up the note.
9. Check whether the day is complete.
10. Save the new `player / camera / agent` state.
11. Synchronize Pixi graphics positions.
