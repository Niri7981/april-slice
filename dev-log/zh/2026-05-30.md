# 2026-05-30 开发日志

今天第一个文件从 `brainTypes.ts` 开始看。

## `brainTypes.ts`

这个文件的职责主要是规定两件事：

1. agent brain 吃进去什么输入？
2. agent brain 必须吐出来什么输出？

几个核心类型：

```text
AgentBrainInput        = 给大脑看的上下文
AgentBrainOutput       = 大脑返回的反应结果
agentBrainOutputSchema = 用 Zod 校验返回结果
```

这个文件把几个系统接起来了：

```text
Initial Hand
    ↓
Agent current state
    ↓
Echo event
    ↓
Agent brain input/output
```

### `agentReactionSchema`

`agentReactionSchema` 规定 agent 能有哪些反应。

也就是说，玩家丢了一个 Echo，agent 不一定直接接受。她可能有 6 种反应：

| reaction | 意思 |
|---|---|
| `accepted` | 接住了 |
| `hesitated` | 犹豫了 |
| `resisted` | 抵抗了 |
| `misread` | 误读了 |
| `delayed` | 延迟反应 |
| `transformed` | 转化成了别的理解 |

### `AgentBrainInput`

`AgentBrainInput` 是喂给大脑的完整上下文。

```ts
export type AgentBrainInput = {
  language: "zh" | "en";
  profile: { ... };
  openingHand: { ... };
  currentState: AgentSignalState;
  relationships: Array<...>;
  dayContext: { ... };
  echoContext: { ... };
  memory: { ... };
  event: AgentBrainEvent;
};
```

这个是文件里最重要的输入类型。agent brain 每次生成反应时，不是只看一句 Echo。

它会看角色设定、开局手牌、当前状态、人际关系、当天上下文、Echo 使用情况、近期记忆和当前事件。

### `agentBrainOutputSchema`

`agentBrainOutputSchema` 规定大脑必须返回什么。

下面这个是输出校验：

```ts
export const agentBrainOutputSchema = z.object({
  behavior: z.object({ ... }),
  diary: z.object({ ... }),
  summary: z.object({ ... }),
  stateChanges: ...,
  relationshipChanges: ...,
  memory: z.object({ ... }),
  meta: z.object({ ... }),
});
```

这个非常重要。最后返回的东西都必须过 schema。

### 数据流向

整体可以这样看：

```text
玩家触发 Echo
    ↓
整理成 AgentBrainEvent

当前角色资料 / Initial Hand / 当前状态 / 关系 / 日期 / 记忆
    ↓
组装成 AgentBrainInput

fakeResolver 或 LLM resolver 读取 AgentBrainInput
    ↓
生成 output

agentBrainOutputSchema.parse(output)
    ↓
得到可信的 AgentBrainOutput
```

然后分发给：

- `behavior` → 世界层表现
- `diary` → 日记 UI / day record
- `stateChanges` → runState / stateDrift
- `relationshipChanges` → relationshipDrift
- `memory.internalThought` → 明天继续喂给 prompt

一句话总结：`brainTypes.ts` 是 agent 大脑层的输入/输出协议文件。

## `echoResolution.ts`

第二个文件看的是 `echoResolution.ts`。

它负责把这几步串起来：

```text
玩家触发 Echo
    ↓
resolveBrain 生成 agent 的反应
    ↓
根据 reaction 计算 stateDrift
    ↓
根据 reaction 计算 relationshipDrift
    ↓
整理成一条 DailyEchoRecord
    ↓
返回完整 EchoResolution
```

这里也能看出一个很重要的边界：LLM 或 fakeResolver 负责生成反应和表达，但状态变化、关系变化和每日记录还是由本地规则整理。


## `initialHandApiClient.ts`

第三个文件是 `initialHandApiClient.ts`。

这个文件本质上是一个 API 客户端函数，主要函数是 `requestInitialHand`。

### 数据流

```text
用户输入 name + birthDate
    ↓
buildAstrologyChartSeed(input)
    ↓
生成 chart seed
    ↓
POST 发给 apiUrl
    ↓
后端返回 InitialHand JSON
    ↓
initialHandSchema.parse(...)
    ↓
前端拿到安全可用的 InitialHand
```

核心代码：

```ts
const response = await fetch(apiUrl, {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    chart: buildAstrologyChartSeed(input),
  }),
});
```

这段的意思是：向 `apiUrl` 发一个 `POST` 请求，把 `chart` 数据用 JSON 格式发给后端。

`JSON.stringify` 的意思是：把对象变成 JSON 字符串。它不是简单套一层 `""`，而是把整个 JS 对象序列化成 JSON 格式。

`Content-Type` 的意思是：告诉后端“我发的是 JSON”。

```ts
"Content-Type": "application/json";
```

也就是说，请求体 `body` 里面的数据类型是 `application/json`。服务器不要把这个 `body` 当普通文本、表单或图片，而是要按 JSON 来解析。

第二个参数：

```ts
{
  method: "POST",
  headers: { ... },
  body: ...
}
```

是一个请求配置对象。所以：

```ts
fetch(apiUrl, { ... })
```

意思是：调用 `fetch` 请求 `apiUrl`，并且用 `{ ... }` 这个对象配置请求方式、请求头和请求体。

## `worker/src/index.ts`

下一个文件是 `index.ts`。

主要作用：接收前端发来的 `chart`，调用模型生成 Initial Hand，然后把模型返回的 JSON 再返回给前端。

### 一句话数据流

```text
前端 requestInitialHand
    ↓
POST /initial-hand
    ↓
index.ts 收到 chart
    ↓
拼 prompt
    ↓
请求 NEWAPI / chat completions
    ↓
拿到模型返回的 JSON 文本
    ↓
extractJsonObject 解析 JSON
    ↓
返回给前端
```

### 核心规则表

| 模块 | 作用 |
|---|---|
| `Env` | 定义环境变量，比如 API key、模型名、允许跨域来源 |
| `RequestRecord` | 记录某个 IP 的请求次数，用来限流 |
| `requestWindowMs = 60_000` | 一个请求窗口是 60 秒 |
| `maxRequestsPerWindow = 20` | 每个 IP 每分钟最多 20 次请求 |
| `maxBodyBytes = 12_000` | 请求体最大 12KB |
| `json(...)` | 快速返回 JSON response |
| `corsHeaders(...)` | 设置跨域请求头，让前端能访问这个 API |
| `isRateLimited(...)` | 判断当前 IP 是否请求太频繁 |
| `extractJsonObject(...)` | 从模型返回文本里抠出 `{...}` 并 `JSON.parse` |
| `fetch(...)` | Worker 主入口，所有请求都从这里进来 |

也就是前端先用：

```ts
buildAstrologyChartSeed(input);
```

生成 `chart`，然后发给这个 API。

如果没有 `chart`，它会返回：

```json
{ "error": "missing_chart" }
```

### 完整流程

```text
用户在前端输入 name + birthDate
    ↓
initialHandApiClient.ts
    ↓
buildAstrologyChartSeed(input)
    ↓
生成 chart seed
    ↓
fetch(apiUrl, {
  method: "POST",
  body: JSON.stringify({ chart })
})
    ↓
POST /initial-hand
    ↓
index.ts 后端收到 chart
    ↓
后端拼 prompt
    ↓
后端请求模型 API：
POST /v1/chat/completions
Authorization: Bearer API_KEY
Content-Type: application/json
body: {
  model,
  messages,
  temperature
}
    ↓
模型返回 Initial Hand JSON 文本
    ↓
后端 extractJsonObject(content)
    ↓
后端把 JSON 返回给前端
    ↓
前端 initialHandSchema.parse(response.json())
    ↓
得到安全可用的 InitialHand
```

中转站主要看这几个东西：

| 字段 | 意思 |
|---|---|
| `model` | 要问哪个模型，比如 `"gpt-5.4"` |
| `messages` | 真正发给模型的对话内容 |
| `temperature` | 模型回答的随机性 / 创造性 |
| `Authorization` | 证明有权限调用这个 API |
| `Content-Type` | 告诉对方发的是 JSON |
